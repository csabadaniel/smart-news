# Copilot Instructions for smart-news

## Commit Convention

When creating commits, always follow the **Conventional Commits** format defined in [CONTRIBUTING.md](CONTRIBUTING.md).

**Quick reference:**
- Format: `<type>: <subject>`
- Use lowercase type prefix (feat, fix, ci, docs, refactor, test, build, chore)
- Imperative mood: "add" not "added"
- No period at end
- Example: `ci: add workflow_dispatch trigger`

For complete guidelines, see [CONTRIBUTING.md](CONTRIBUTING.md).

## Testing Approach: London School TDD

This project follows **London School Test-Driven Development** for all implementation work.

### Key Principles
- **Always write tests first** before implementing any feature
- **Use mocks and stubs** to isolate the unit under test from external dependencies
- **Focus on behavior**, not implementation details
- **Outside-in approach**: Start with public APIs and interfaces, then implement the underlying logic
- **Tests as specifications**: Treat tests as executable documentation of expected behavior

### Implementation Workflow
1. Write a failing test that describes the desired behavior
2. Implement the minimal code to make the test pass
3. Refactor for clarity and maintainability
4. Move to the next test

### Mocking Guidelines
- Mock all external dependencies: APIs, services, repositories, configuration
- Use dependency injection to make services testable
- Focus unit tests on the service's own logic, not on external systems
- Integration tests (if needed) should target specific cross-system interactions

### Test Naming Convention
- Use simple, descriptive names in imperative mood
- Examples: `shouldGenerateNewsFromValidPrompt`, `throwsExceptionForNullInput`, `returnsEmptyListWhenNoResults`
- Omit the test method logic from the name; focus on the behavior being tested
- Keep names readable and reasonably concise

For complete testing guidelines, see the "Testing Strategy" section in [CONTRIBUTING.md](CONTRIBUTING.md).
